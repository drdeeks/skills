## Where the rules live

**Nowhere in this skill.** skill-creator's `scripts/validate.py` is the
single source of truth; the installer delegates every verdict to it via
`scripts/validate_install.py` (see [creator-delegation.md](creator-delegation.md)).
This document is a convenience summary so an operator can predict outcomes —
if it ever disagrees with skill-creator's validator, the validator wins and
this summary is what gets corrected.

## Validation pipeline

```
Input → Format Detection → Extraction → Validation → Installation → Post-Check
                              ↓              ↓
                         skill-creator  skill-creator
                          validate.py    validate.py
```

Installs gate at **basic** tier by default; `--enterprise` gates harder.
If skill-creator cannot be located, the install is refused — the installer
never proceeds unvalidated.

## Current standard (summary)

Exactly **5 root items** per skill, both tiers:

```
skill-name/
├── __init__.py         (functional — scaffolds the skill's own domain)
├── SKILL.md            (frontmatter: name, description, version; body ≤500 lines)
├── scripts/            (flat; script extensions only)
├── references/         (.md/.txt/.html/.pdf; optional lessons/ + templates/ subdirs)
└── <skill_name>.skill  (regenerated by skill-creator's packager)
```

- `assets/` is banned anywhere in a skill
- `references/templates/` is the only place for templates: read-only, any
  file type, purpose-scoped names
- No generic file names anywhere (readme/notes/untitled/numbered docs)
- `version` in frontmatter is mandatory
- Guardrail artifacts (`.gate.json`, `.loop-log.jsonl`, etc.) at a skill
  root are opaque — never counted, moved, or flagged

## Tier gates

| Gate | Basic | Enterprise |
|------|-------|------------|
| metadata.tags | 5+ | 7+ |
| Substantive scripts | 2+ | 3+ |
| Substantive references | 3+ | 5+ |
| Structure + content scans | identical | identical |

## Content scans

All of these apply to **every file** in the skill, not just SKILL.md:

- **Placeholders** — work markers (`TODO`-class words), filler text, and
  template variables are rejected; fenced code blocks and inline code
  spans are treated as quoted material, not live markers
- **Hardcoded secrets** — literal credential assignments in scripts reject
- **Hardcoded paths** — home/root/opt/temp path literals reject
- **Nested SKILL.md** — only the root SKILL.md is allowed
- **Broken internal links** — SKILL.md links must resolve
- **Duplicate sections** — repeated `##` headings reject

There is no scoring or leniency system: the verdict is `fails == 0` at the
requested tier. Warnings are advisory and never block.

## Verdict format

`validate_install.py --json` relays skill-creator's verdict:

```json
{
  "operation": "validate_install",
  "success": true,
  "delegated_to": "<skill-creator>/scripts/validate.py",
  "tier": "basic",
  "valid": true,
  "status": "pass",
  "fails": 0,
  "warnings": 1,
  "checks": [ {"name": "...", "severity": "FAIL", "passed": true, "detail": "..."} ]
}
```

`success: false` means the delegation itself failed (skill-creator missing,
timeout, bad output) — treated as a rejection, never as a pass.
