# HMAC Audit Log Reference

Every successful gate run appends one line to the loop log (`.loop-log.jsonl`).
Each line is both **signed** (HMAC-SHA256 under the repo secret) and **chained**
(each entry commits to the previous one's hash), giving a tamper-evident audit
trail: you cannot forge, insert, delete, or reorder entries without the secret.

## Entry Shape

```json
{
  "timestamp": "2026-07-02T14:30:00Z",
  "gated_dir": "/path/to/dir",
  "variables": {"path": "./output/proj"},
  "result": "PASS",
  "prev_hash": "<sha256 of the previous entry, or 64 zeros for the first>",
  "entry_hash": "<sha256(prev_hash + canonical_body)>",
  "hmac_sha256": "<hmac_sha256(secret, entry_hash)>"
}
```

## How the Chain Is Built (`gate.py`)

1. Read the `entry_hash` of the last log line (or the genesis constant, 64 zeros,
   for an empty log).
2. Build the entry body, set `prev_hash` to that value.
3. `entry_hash = sha256(prev_hash + canonical_json(body))`, where canonical JSON
   is sorted-key, comma/colon-separated — deterministic across machines.
4. `hmac_sha256 = HMAC-SHA256(secret, entry_hash)`.
5. Append the fully-populated object as one JSON line.

Because `entry_hash` folds in `prev_hash`, changing any earlier entry changes
every later `entry_hash`, and each of those has a signature that will no longer
verify. A single byte flip anywhere breaks the whole tail.

## Verification (`verify_log.py`)

```bash
# Full integrity check (signatures + chain)
python3 scripts/verify_log.py .loop-log.jsonl --secret ~/.config/gate/hmac.key

# Report every defect rather than stopping at the first
python3 scripts/verify_log.py .loop-log.jsonl --all

# Git-hook mode: also require a PASS entry newer than N seconds
python3 scripts/verify_log.py .loop-log.jsonl --recent 3600
```

Verification recomputes each `entry_hash` from the body and prior hash, checks it
matches, then checks the HMAC with a constant-time compare. Exit 0 only when the
entire log verifies.

## The Secret

- Generated by `setup.py` as 256 bits of `secrets.token_hex(32)`, stored `0600`.
- Default location `~/.config/gate/hmac.key` — per user, never inside the skill
  archive, never committed. A `.skill` you ship therefore contains **no** secret
  and cannot sign or verify anyone else's log; recipients generate their own.
- Losing the secret means the existing log can no longer be verified — treat it
  like any signing key. Back it up out-of-band if the audit trail must survive a
  machine loss.

## Parked Design Questions

The following were deliberately left open in the CL-040 design (see
[proposed-design.md](proposed-design.md)) and are worth revisiting before a
multi-user rollout:

- **Rotation** — supporting multiple valid keys during a rotation window so old
  entries still verify while new ones use the new key.
- **Cross-log tamper detection** — the chain already detects tampering within one
  log; a Merkle-root checkpoint published elsewhere would detect wholesale log
  replacement.
